# 🚀 코드 리팩토링 및 최적화 가이드 (Universal Guide)

본 문서는 소프트웨어의 가독성, 유지보수성, 성능을 극대화하기 위한 범용적 리팩토링 원칙을 정의합니다. 모든 개발 단계에서 **"단순함"**과 **"검증"**을 최우선 가치로 삼습니다.

---

## 1. 과잉 엔지니어링(Over-Engineering) 방지

가장 뛰어난 설계는 **"현재의 문제를 해결하는 가장 단순한 구조"**입니다. 미래를 대비한 복잡한 설계는 오히려 독이 될 수 있습니다.

| 원칙 | 상세 내용 |
|------|-----------|
| **YAGNI** | You Ain't Gonna Need It. "나중에 필요할 것 같아서" 미리 만드는 기능은 90% 확률로 기술 부채가 됨. |
| **KISS** | Keep It Simple, Stupid. 구조를 설명하기 위해 긴 문서가 필요하다면 실패한 설계임. 로직은 직관적이어야 함. |
| **모듈화의 균형** | 과도한 분리는 모듈 간 통신 복잡도와 보일러플레이트만 증가시킴. 실질적 재사용성이 확인될 때 분리할 것. |
| **의존성 최소화** | 특정 모듈의 변경이 다른 모듈로 전파되지 않도록 인터페이스 경계를 명확히 설정함. |

---

## 2. 리팩토링 핵심 원칙 (Clean Code)

### 🔹 의미 있는 명명 (Naming)

변수/함수명만으로 **역할(What)**과 **이유(Why)**가 드러나야 함.

- `data`, `handle`, `temp` 등 모호한 단어 지양.
- `buffer_index`, `validate_sensor_input` 등 구체적 명칭 사용.

### 🔹 실용적 단일 책임 (Pragmatic SRP)

하나의 단위(함수/클래스)는 하나의 논리적 역할만 수행함.

- 단, "한 줄 한 책임" 수준의 파편화는 지양하고, 전체 맥락 파악이 용이한 수준에서 분리함.

### 🔹 추상화 수준 통일

같은 계층(파일/함수) 내에서는 비슷한 수준의 추상화만 유지함.

- 고수준 비즈니스 로직과 저수준 비트 연산/문자열 파싱이 섞이지 않도록 함.

### 🔹 하드코딩 제거 (Magic Values)

모든 설정값, 경로, 상수는 별도의 `config` 또는 `constants` 파일로 관리함.

- 코드 내부에 생으로 적힌 숫자나 문자열은 반드시 의미 있는 이름의 상수로 치환함.

## 3. 구조 및 성능 최적화 전략

### 📂 구조 최적화

- **계층 최소화:** 3~4단계 이상의 깊은 폴더 구조는 지양하고 역할별로 평탄하게 유지함.
- **선 병합, 후 분리:** 처음부터 쪼개지 말고, 파일이 너무 커지거나(예: 800라인 이상) 명확한 재사용 포인트가 보일 때만 분리함.
- **Dead Code 제거:** 미사용 변수, 호출되지 않는 함수, 주석 처리된 옛 코드는 즉시 삭제함.

### ⚡ 성능 및 리소스 최적화

- **지연 로딩 (Lazy Loading):** 대규모 에셋이나 라이브러리는 프로그램 시작 시점이 아닌, 실제 필요 시점에 로드함.
- **루프 최적화:** 반복문 내부에서 변하지 않는 연산은 외부로 추출하고, 빈번한 I/O 호출을 최소화함.
- **메모리 관리:** 불필요한 객체 생성을 줄이고, 대량의 데이터 처리 시 배치(Batch) 처리를 고려함.

## 4. 리팩토링 진행 프로세스

리팩토링은 기능을 추가하는 것이 아니라 내부 구조를 개선하는 과정입니다. 반드시 다음 단계를 준수합니다.

1. **현상 파악:** 중복 코드, 복잡한 조건문, 성능 저하 지점 식별.
2. **검증 수단 확보:** 리팩토링 전후 결과가 동일함을 증명할 테스트 환경 확인.
3. **최소 단위 실행:** 한 번에 전체를 고치지 말고, 함수 단위 혹은 클래스 단위로 한 블록씩 수정.
4. **중간 검증:** 수정 직후 즉시 실행하여 사이드 이펙트 여부 확인.
5. **문서/주석 갱신:** 변경된 구조에 맞게 API 명세나 관련 문서 최신화.

**진행 시 참고**

- **커밋:** `refactor:`, `perf:` 접두사 사용 권장.
- **에이전트/협업 시 예:** "이 지침의 [단계 N] 실행해줘", "체크리스트 기준으로 [범위] 검토해줘" 등으로 지침을 지정할 수 있음.

---

## 5. 최종 체크리스트 (Definition of Done)

- [ ] **가독성:** 코드만 보고도 로직의 흐름을 1분 내에 파악할 수 있는가?
- [ ] **중복:** 동일하거나 유사한 로직이 3곳 이상 반복되고 있지 않은가?
- [ ] **안정성:** 리팩토링 후 기존 기능(LOAD → RUN → EXIT 등)이 정상 작동하는가?
- [ ] **규격:** config나 상수가 적절히 분리되어 관리되는가?
- [ ] **깔끔함:** 사용하지 않는 import, 주석 처리된 코드, 쓰지 않는 변수가 모두 제거되었는가?

---

## 6. 프로젝트에 맞게 채울 항목 (선택)

다른 프로젝트에 이 문서를 가져다 쓸 때, 아래만 채우면 범위와 성공 기준을 명확히 할 수 있습니다.

- **범위:** 리팩토링 대상 경로 (예: `src/`, `packages/foo/`).
- **성공 기준:** 빌드 통과, 테스트 통과, 특정 시나리오 동작(LOAD → RUN → EXIT 등), 리로드 안정성 등.
- **참조 문서:** PRD, 아키텍처 스펙, API 스펙 등 (있을 경우).

위 항목을 채우지 않아도 이 지침서는 그대로 사용 가능합니다.

---

**Note:** 본 지침은 프로젝트의 성격에 따라 유연하게 조정될 수 있으나, **"단순함이 복잡함보다 낫다"**는 원칙은 변하지 않습니다.
