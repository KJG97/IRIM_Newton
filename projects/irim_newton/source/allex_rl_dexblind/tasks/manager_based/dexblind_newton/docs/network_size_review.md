# Dexblind Newton PPO 네트워크 크기 검토

## 1. 현재 설정 (LSTM=True)

| 구분 | 구조 | 입력 차원 | 비고 |
|------|------|-----------|------|
| **Actor** | LSTM(1024) → MLP [1024, 512, 256, 128] | **~345** | policy(90) + proprio(255), history=5 |
| **Critic** | MLP [1024, 512, 256, 128] | **~95** | policy_critic(18) + privileged(77) |
| **Action** | — | **18** | arm 7 + hand 11 |

- Policy obs: `actions` 18×5 = 90
- Proprio obs: (joint_pos 18 + torque 19 + hand_pose 7 + hammer_initial 7)×5 = 255
- Critic obs: 18 + (18+19+7+7+3+3+3+3+1+1+15+1+1+1) = 95

## 2. 레포 내 유사 태스크 비교

| 태스크 | Actor/Critic hidden | 비고 |
|--------|---------------------|------|
| **dexblind** (동일 패밀리) | [512, 256, 128] | LSTM 없음 |
| dexblind chunked | [512, 512, 256] | 3층 |
| direct/allex | [256, 128, 64] | 더 단순 |
| **Allegro hand** (손만) | [1024, 512, 256, 128] | 고차원 손 제어 |
| Humanoid / Ant | [400, 200, 100] | 3층 |
| Locomotion (Go2, H1 등) | [512, 256, 128] 또는 PLAY [128,128,128] | 3층 |
| Reach (Franka/UR10) | [64, 64] | 단순 도달 |

## 3. 결론: 현재 설정은 태스크 대비 과한 편

### Actor
- **입력 345차원 + LSTM**으로 시퀀스 정보는 이미 충분히 다룸.
- 그 위에 **4층 MLP [1024→512→256→128]**는 같은 계열인 **dexblind**([512,256,128])보다 훨씬 큼.
- Allegro hand와 동일한 MLP를 쓰고 있지만, Allegro는 손만 제어하고 여기서는 arm+hand+망치 리프트로 목표가 더 구체적임.
- **권장**: LSTM 유지 시 MLP는 **[512, 256, 128]**로 축소 (dexblind와 동일). LSTM hidden은 **512**로 줄여도 시도해볼 만함.

### Critic (과다가 가장 큼)
- **입력 95차원**인데 **첫 층 1024**면 약 10배 확장으로 과함.
- Value는 스칼라 1개만 예측하므로 같은 레포의 작은 설정들처럼 더 작은 MLP로 충분.
- **권장**: **[256, 256, 128]** 또는 **[512, 256, 128]** (입력 대비 2~5배 수준).

### 요약
- **Actor**: [1024, 512, 256, 128] → **[512, 256, 128]** 권장 (선택: LSTM 1024→512).
- **Critic**: [1024, 512, 256, 128] → **[256, 256, 128]** 또는 **[512, 256, 128]** 권장.

이렇게 줄이면 파라미터 수·연산량이 줄어 학습 속도와 안정성에 도움이 될 수 있고, 성능은 비슷하거나 오히려 나을 수 있음.
